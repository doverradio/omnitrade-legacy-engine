# SECURITY_AND_SAFETY.md

## OmniTrade Legacy Engine — Security & Safety Rules

These rules apply to every phase, every environment, and every contributor (human or Copilot-assisted). They are not suggestions — a PR that violates any rule below should not be merged.

### 1. No Live Trading

- The MVP contains **no code path** capable of placing a real, capital-at-risk order.
- Alpaca integration uses only the paper trading base URL (`https://paper-api.alpaca.markets`), configured via `ALPACA_BASE_URL` in `app/config.py`. This value must never be overridden to point at Alpaca's live trading endpoint anywhere in MVP code, config, or environment templates.
- Crypto execution is entirely internal simulation (`app/services/paper/internal_sim.py`) against Binance/Binance.US market data — no order-placement calls are ever made to Binance/Binance.US.
- `trades.is_paper` is `true` for every row created in MVP; any code that would set it `false` is out of scope and must not be written.
- Enabling live trading in the future requires a separate, explicitly-approved initiative per `RISK_ENGINE.md` §5 — not a configuration toggle that can be flipped casually.

### 2. No Real API Keys Committed

- `.env`, `.env.local`, and any file matching `*.env` (except `*.env.example`) are in `.gitignore` from Phase 0 onward — verify this is never removed.
- Only `infra/env-templates/*.env.example` and the root `.env.example` are committed, and they contain placeholder values only (e.g., `ALPACA_API_KEY_ID=your_paper_key_here`), never real credentials.
- Before every commit/PR, run a secrets scan (e.g., `git diff` review, or a pre-commit hook using a tool like `detect-secrets` or `gitleaks` once CI is set up in Phase 8) to catch accidentally staged credentials.
- If a real secret is ever accidentally committed: treat it as compromised immediately — rotate the credential at the provider (Alpaca/Binance/Supabase) rather than relying solely on removing it from history.
- Alpaca keys used anywhere in this project (local, staging, "production") are **paper trading keys only** — never generate or use Alpaca live-trading keys for this project during MVP.

### 3. Paper Trading Only (Environment Labeling)

- Every environment — including whatever is deployed as "production" in Phase 8 — operates in paper-trading mode. There is no separate "real" production in MVP.
- The frontend `PageShell` top bar always displays the current environment name and a persistent "PAPER TRADING" label, per `UI_SPEC.md` §1 and `FRONTEND_PAGE_SPECS.md` cross-page requirements — this must never be hideable or removable by a user-facing setting.
- Any explanation, report, or exported data generated by the system must avoid language that could be mistaken for real brokerage/exchange statements (e.g., always say "simulated," "paper," or "backtested," never bare "profit" or "account balance" without that qualifier in user-facing copy).

### 4. Kill Switch Architecture

- Two independent kill switch levels exist: **global** (halts all trading platform-wide) and **account-level** (halts one paper account). See `RISK_ENGINE.md` §2.8 for full behavior.
- Kill switch state is persisted in the database (not just in-memory), so a service restart does not silently clear an active kill switch.
- Automatic triggers (repeated data/reconciliation errors, repeated risk engine internal errors) are implemented as a documented, testable rule set in `app/services/risk/rules/kill_switch.py` — not ad hoc exception handling scattered through the codebase.
- Re-arming a kill switch always requires an explicit, authenticated human action, logged to `audit_log` with the actor identity — no automatic or scheduled re-arm logic is permitted.
- The signal-generation loop (`app/services/signals/`) must check kill switch state as its very first step, before any strategy/AI computation runs, so a tripped kill switch has zero chance of being bypassed by a downstream code path.

### 5. Rate Limiting

- All outbound calls to Binance/Binance.US and Alpaca go through the shared `app/services/data/http_client.py` backoff wrapper (`COPILOT_PHASE_1_PROMPTS.md` Prompt 1.3) — no service is permitted to call these APIs directly with a raw HTTP client.
- Inbound API rate limiting (protecting the FastAPI backend itself) is out of scope for a single-family MVP but should be added via standard middleware (e.g., `slowapi`) before any multi-user/public deployment beyond the immediate family use case.
- Scheduled workers (ingestion, signal generation) run on conservative intervals (see `COPILOT_PHASE_1_PROMPTS.md` Prompt 1.9 — 5-minute default) specifically to stay well within documented exchange/broker rate limits, not just to hit them and rely on backoff.

### 6. Secrets Management

- All secrets are environment variables only, loaded exclusively through `app/config.py`'s `Settings` class (backend) and Next.js's `NEXT_PUBLIC_*` convention (frontend, for the small set of values safe to expose client-side — never API keys).
- Local: `.env` / `.env.local` files (gitignored).
- Staging/Production: platform-native secret managers (Vercel environment variables for frontend; Railway/Fly.io/Render environment variables for backend/workers; Supabase project settings for DB/Auth keys) — never files committed to the deploy target's filesystem via the repo.
- No secret is ever logged. `app/core/logging.py` configuration and all service code must avoid logging full request/response bodies from Alpaca/Binance calls that could contain key material — log status codes, error messages, and non-sensitive metadata only.
- Service-role/admin-level Supabase keys are backend-only and never sent to or referenced by `apps/web`.

### 7. Audit Logging

- Every state-changing action writes an `audit_log` row per `DATABASE_SCHEMA.md` §2.12: parameter changes, strategy activation/deactivation, backtest runs, trades, risk events, kill switch trips/re-arms, and paper account resets.
- Audit log rows are never updated or deleted through any application code path — no `PATCH`/`DELETE` route exists for `audit_log` in `API_CONTRACTS.md`, and this must remain true as the API surface grows.
- Every AI decision additionally writes to `model_outputs` (per `AI_LAYER.md` §6) with enough input context to reproduce why a given output was produced.
- The `/settings` page's `AuditLogViewer` is read-only by design (`FRONTEND_PAGE_SPECS.md`) — no UI path to edit or clear audit history.

### 8. Disclaimers

The following disclaimer language (or equivalent) must appear in the application UI (e.g., footer or an about/settings section) and in this documentation set's user-facing surfaces:

> "OmniTrade Legacy Engine is a research and paper-trading tool. It does not execute real trades, does not manage real money, and does not provide financial, investment, tax, or legal advice. Past backtest or paper-trading performance does not guarantee future results. No output from this system — including AI-generated confidence scores, regime classifications, or explanations — should be construed as a recommendation to buy or sell any real security or asset."

- This disclaimer is a baseline, not a substitute for actual legal review before any future live-trading initiative or any use beyond the immediate family/personal research context described in `PROJECT_VISION.md`.
- No feature, prompt, or generated explanation may claim or imply guaranteed profitability, "guaranteed win rate," or similar — this is both an ethical commitment (`PROJECT_VISION.md` §4) and a practical safeguard against misleading outputs.

### 9. Contributor Checklist (Human or Copilot-Generated Code)

Before merging any PR, confirm:
- [ ] No real credentials anywhere in the diff.
- [ ] No new code path reaches a live-trading endpoint.
- [ ] Any new state-changing endpoint writes an `audit_log` row.
- [ ] Any new AI-generated output writes a `model_outputs` row with a non-empty explanation.
- [ ] Any new destructive UI action has a confirmation step.
- [ ] Rate-limited/backoff HTTP client is used for any new external API integration.
